# 90 分ワーク — AI 駆動開発サイクルを体験する

プログラミング経験者を対象に、Claude Code を使った **「タスク整理 → コード理解 → 計画 → 承認 → 実装 → レビュー → PR」** までの一連のサイクルを 90 分で体験するワークショップです。最後に「2030 年のエンジニア像」ミニ講義＋意見交換を行います。

題材は [quickstart.md](../tutorial/quickstart.md) と同じく、[主催者がイベントを公開する](./issues-20260419.md#item-主催者がイベントを公開する) シナリオ (event BC / PublishEvent) を使います。

---

## タイムテーブル (90 分)

| 時間 | セクション | 主な活動 |
|-----:|-----------|---------|
| 00–10 | 1. タスク整理 | AI と対話してチケットを切る |
| 10–20 | 2. コード理解 | 既存構造を AI に質問しながら学ぶ |
| 20–35 | 3. 計画 | AI に実装計画を立ててもらう |
| 35–45 | 4. 承認 (意思決定) | **人が落ち着いて読む** / モブで議論 |
| 45–65 | 5. 実装 (しばらく動かす) | AI に書かせる＋テスト |
| 65–75 | 6. コードレビュー | AI レビュー＋人の最終チェック |
| 75–80 | 7. プルリクを投げる | `/japanese-commit` → PR |
| 80–90 | 8. ミニ講義＋意見交換 | 2030 年のエンジニア像 |

## 1. タスク整理 (10 分)

**ゴール**: 曖昧な要望を AI と対話しながらチケット粒度に落とす。

### やること

```
@docs/issues-20260419.mdを元に イシューを作成して
```


### ポイント

- AI に **インタビュアー役** を依頼する — 「不明点があれば質問して」と添えると考慮漏れを先に潰せる
- 受入基準 (AC) が 1 テスト 1 項目に分解できる粒度まで落とす
- 依存関係 (前提となる BC / データ) を洗い出す

---

## 2. コード理解 (10 分)

**ゴール**: 既存コードベースの構造とパターンを掴む。

### やること

Plan Mode (Shift+Tab) に切り替えてから:

```
backend の community / event コンテキストのレイヤー構造を教えて。
CreateEvent はどう実装されている？
```

```
@AGENTS.mdに記載されている、バックエンドの実装パターン  を要点解説して
```

### 確認ポイント

- `controllers/` → `usecases/` → `models/` のレイヤー構造
- `Result<T, E>` / Branded Types / Zod スキーマの存在
- `CreateEvent` の参照実装箇所

---

## 3. AI に計画を立ててもらう (15 分)

**ゴール**: 実装前に What + Why + 参考パターンで AI と計画を合意する。

### やること

プランモード
```
issues#7 を実装したい。event BC の CreateEvent 一式をパターン参照して、
backend に絞って実装計画を立てて。疑問点あれば質問して
```

### 計画に含めるべきもの

| レイヤー | 内容 |
|---------|------|
| Model | `publishEvent()` (DRAFT → PUBLISHED) を `models/event.ts` に追加 |
| UseCase | `publish-event.command.ts` (Result<Event, PublishEventError>) |
| Controller | PUT エンドポイント＋ `event-openapi.ts` 更新 |
| Error | `PublishEventError` (`EventAlreadyPublished` / `Unauthorized` / `EventNotFound`) |
| Event | `EventPublished` ドメインイベントの publish (任意) |

---

## 4. 承認 — 意思決定 (10 分) 【このワークの核】

**ゴール**: AI の計画を **人が落ち着いて読み込み**、納得してから承認する。

> **なぜここが重要か**
> 近年「AI が書けること」より **「何を作るかを決める意思決定」** のほうが価値を持つと言われています。**AI-DLC** で代表されるように、モブプログラミング的に **多様なメンバーで複数人で意思決定** することで、後戻りが減り品質が上がります。
> AI が書いたコードを後から直すコストより、計画段階で 5 分議論するほうが圧倒的に安い。

### レビュー観点

- [ ] レイヤー配置が正しいか (ドメインロジックが UseCase に漏れていないか)
- [ ] ファイル名・配置が既存パターンと揃っているか
- [ ] 受入基準とテストケースが 1:1 対応しているか
- [ ] エラー型の粒度が妥当か

### モブでの進め方

- 1 人が画面共有し、**声に出して計画を読む**
- 違和感があれば止めて議論
- 気になる箇所は `Ctrl+G` で計画をエディタで開いて直接編集
- 合意が取れたら承認

---

## 5. 実装 — しばらく動かす (20 分)

**ゴール**: 承認済み計画を AI に実行させ、テストまで通す。

### やること

Plan Mode なら承認でそのまま走る。通常モードなら:

```
OK、この計画で実装して。テストも書いて
```

### 途中で監視するポイント

- **計画外のファイルを触り始めたら Esc で止める**
- 2 回修正しても直らなければ `/clear` して再出発
- 出力が安定しないときは依頼を小さく切る (model だけ → UseCase だけ)

### テストが通ったあと

テスト通過 ≠ 正しい実装。**テストコード自体を人が読む**:

- 受入基準を正しく表現しているか
- 境界ケース (既に PUBLISHED / 作成者以外) が網羅されているか
- モックが現実の Repository 振る舞いと乖離していないか

## 8. ミニ講義＋意見交換 (10 分)

### 2030 年のエンジニア像

- **書く人から決める人へ** — コードを書くより「何を・なぜ作るか」「どこで止めるか」の意思決定が主戦場になる
- **AI-DLC 的な協働** — 多様なメンバー (PdM / デザイナー / QA / ドメインエキスパート) と AI が同じ「場」でモブする
- **ハーネスを設計する力** — 個別タスクを解くより、AI が正しく動く環境 (規約・自動検証・参照実装) を整える人が価値を持つ
- **ドメイン理解と検証設計** — テストコードを書く能力と、ドメインを言語化する能力が差別化要因に

### 意見交換のテーマ例

- 今日のワークで **一番コストが高かった / 価値が高かった** のはどこか？
- 自分のチームに持ち帰るなら、まずどこから始めるか？
- 「人が決めるべきこと」と「AI に任せていいこと」の境界線は？
