# モノレポ構成の採用

日付: 2026-04-04
種類: arch
ステータス: 承認済み

## コンテキスト

Meetup コミュニティ管理アプリケーションは、Express.js バックエンドと React フロントエンド、Playwright E2E テストで構成されるフルスタックアプリケーションである。これらのコードベースをどのように管理するかを決定する必要があった。

主な要件:
- バックエンドとフロントエンドで型定義やスキーマを共有したい
- 開発環境のセットアップを簡素化したい
- CI/CD パイプラインを統一的に管理したい
- E2E テストがバックエンドとフロントエンドの両方に依存する

## 決定

npm workspaces を使用したモノレポ構成を採用する。ルート `package.json` で `backend/`, `frontend/`, `e2e/` の3ワークスペースを管理する。

## 理由と代替案

**採用した理由:**
- バックエンドとフロントエンド間で型定義・スキーマの共有が容易（パッケージ公開不要）
- 単一の `docker compose` + ラッパースクリプトで開発環境を一括管理できる
- E2E テストがバックエンド・フロントエンド両方のソースに隣接し、変更の整合性を保ちやすい
- npm workspaces はNode.js 標準機能であり、追加ツール不要で依存関係を一元管理できる
- git worktree と組み合わせてフィーチャーブランチの並行開発が可能

**検討した代替案:**
- **マルチレポ（backend / frontend / e2e を別リポジトリ）**: リポジトリ単位の独立性は高い → 選ばなかった理由: 型定義の共有にパッケージ公開が必要になり、開発サイクルが遅くなる。E2E テストの変更と本体の変更をアトミックにコミットできず、バージョン不整合のリスクが高い
- **Turborepo / Nx 等のモノレポツール**: ビルドキャッシュやタスクグラフによる高度な最適化が可能 → 選ばなかった理由: 現時点でワークスペースが3つと小規模であり、npm workspaces の標準機能で十分。ツール固有の設定・学習コストに見合うメリットがまだない。規模が拡大した場合に導入を再検討する
- **フロントエンドをバックエンドに同梱（単一パッケージ）**: 構成が最もシンプル → 選ばなかった理由: DDD のコンテキスト分離（auth / meetup）をバックエンド・フロントエンド双方で維持しており、単一パッケージでは責務の境界が曖昧になる。テスト実行やビルドの独立性も失われる

## 影響

**ポジティブ:**
- 型安全性の向上: バックエンドの Zod スキーマから推論した型をフロントエンドでも参照可能
- 開発体験の統一: `d dev` で全サービスが起動、`d test` で全テストが実行
- 変更のアトミック性: API 変更とフロントエンド対応を同一コミット・同一 PR で管理できる
- Docker 構成の簡素化: 単一の `docker-compose.yml` でモノレポ全体をマウント

**ネガティブ・リスク:**
- リポジトリサイズの増大: バックエンド・フロントエンドの依存パッケージが同一 `node_modules` に集約される
- CI の実行時間: 変更がなくても全ワークスペースのテストが走る（将来的にタスクグラフツールで最適化可能）
- ワークスペース間の暗黙的結合: 共有が容易なぶん、意図しない依存が生まれるリスクがある
