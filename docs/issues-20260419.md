# ミートアップサンプル v2 — イシューバックログ (2026-04-19)

> **本ドキュメントは 2026-04-19 時点の「ハンズオン出発点」バックログ。** community BC と event BC の `CreateEvent` までを実装済みとし、それ以降の SCENARIO / POLICY は `status:todo` として未実装化した。完全実装版のコードと実装手順の参考は [`snapshot/2026-04-19`](https://github.com/haru01/meetup-sample-v2/tree/snapshot/2026-04-19) ブランチで保全済み。

## 概要

- 元ドキュメント: [docs/eventstorming-20260417-2009.md](eventstorming-20260417-2009.md)
- 3 層構造: **context (BC) → feature (業務フロー) → item (SCENARIO / POLICY)**
- 階層はチケット登録時に **ラベル** で表現する
- ラベル設計:
  - 階層: `context:<bc>` / `feature:<flow>`
  - 種別: `type:scenario` / `type:policy`
  - 状態: `status:done`（実装・テスト完了）／`status:todo`（未着手）
- 件数: 29 items（community 5 / event 9 / participation 8 / checkin 1 / notification 6）
- 状態内訳: `status:done` = 6（community 5 + CreateEvent 1）/ `status:todo` = 23
- 各 item は backend + frontend を端から端まで実装する単位（必要に応じて UI 画面の実装も受入基準に含める）
- **実装済み範囲**:
  - community BC: `CreateCommunity` / `JoinCommunity` / `ApproveMember` / `RejectMember` / `LeaveCommunity`
  - event BC: `CreateEvent`（DRAFT 作成 + コミュニティ別イベント一覧表示のみ）
- **未実装範囲**: `PublishEvent` 以降の event 全シナリオと POLICY、participation / checkin / notification BC 全体
- 参考実装は `snapshot/2026-04-19` ブランチ参照。以前存在した `status:refactor` item（notification BC の 4 通知 UseCase 抽出）は、抽出元の inline 実装自体が未実装に戻ったため `status:todo` に統一

## 凡例

各 item は以下のテンプレートで記述する。

```markdown
#### [item] <日本語タイトル>
- **Labels**: `context:<bc>` `feature:<flow>` `type:<scenario|policy>` `status:<done|todo>`
- **Who**: <アクター>
- **What**: <何を行うか>
- **Why**: <ビジネス価値・ハッピーパス上の位置付け>
- **受け入れ基準**:
  - [x/ ] 条件 1（done は `[x]`、todo は `[ ]`）
- **DML**: SCENARIO / POLICY 名、関連 EVT / CMD / QRY
- **関連コード** or **参考実装 (snapshot/2026-04-19)**: 実装ファイルパス
```

## Context: community

### Feature: コミュニティ管理

#### [item] 主催者がコミュニティを作成する
- **Labels**: `context:community` `feature:community-admin` `type:scenario` `status:done`
- **Who**: 主催者（Organizer）
- **What**: 名前・説明・カテゴリ・公開設定を指定してコミュニティを作成する。作成者を OWNER として同時登録する。
- **Why**: 主催者がイベント開催の母体となるコミュニティを立ち上げる起点となる。
- **受け入れ基準**:
  - [x] 名前が 1〜100 文字で指定できる
  - [x] 説明は 1000 文字以内・nullable である
  - [x] カテゴリは `TECH` / `BUSINESS` / `HOBBY` のいずれかである
  - [x] 公開設定は `PUBLIC` / `PRIVATE` のいずれかである
  - [x] システム全体で名前が重複する場合 `DuplicateCommunityNameError` を返す
  - [x] 作成者が OWNER ロール・ACTIVE ステータスの `CommunityMember` として同時に作成される
  - [x] `CommunityCreated` が発火される
- **DML**: SCENARIO `主催者がコミュニティを作成する`, CMD `CreateCommunity`, EVT `CommunityCreated`
- **関連コード**: `backend/src/community/usecases/commands/create-community.command.ts`

### Feature: メンバーシップ管理

#### [item] メンバーがコミュニティに参加する
- **Labels**: `context:community` `feature:membership` `type:scenario` `status:done`
- **Who**: メンバー（Member）
- **What**: 対象コミュニティへの加入を申し込む。公開設定に応じて即 ACTIVE / PENDING を分岐する。
- **Why**: PUBLIC コミュニティは摩擦なく参加でき、PRIVATE は主催者の承認制にしてコミュニティの質を担保する。
- **受け入れ基準**:
  - [x] 既に同じコミュニティのメンバーの場合 `AlreadyMemberError` を返す
  - [x] `visibility = PUBLIC` のとき ACTIVE で登録され `MemberJoined` が発火される
  - [x] `visibility = PRIVATE` のとき PENDING で登録され `MemberApplicationSubmitted` が発火される
  - [x] 同一 `(communityId, accountId)` で複数レコードが作成されない
- **DML**: SCENARIO `メンバーがコミュニティに参加する`, CMD `JoinCommunity`, EVT `MemberJoined` / `MemberApplicationSubmitted`（WHEN 分岐）
- **関連コード**: `backend/src/community/usecases/commands/join-community.command.ts`

#### [item] 主催者がメンバー申請を承認する
- **Labels**: `context:community` `feature:membership` `type:scenario` `status:done`
- **Who**: 主催者（Organizer）
- **What**: PENDING 状態の加入申請を ACTIVE に昇格する。
- **Why**: PRIVATE コミュニティの主催者が、参加者の適合性を確認してから迎え入れる運用を可能にする。
- **受け入れ基準**:
  - [x] 対象が PENDING 状態のときのみ承認できる
  - [x] PENDING 以外の状態で呼ばれた場合 `MemberAlreadyActiveError` を返す
  - [x] 承認後は `status=ACTIVE` に遷移する
  - [x] `MemberApproved` が発火される
- **DML**: SCENARIO `主催者がメンバー申請を承認する`, CMD `ApproveMember`, EVT `MemberApproved`
- **関連コード**: `backend/src/community/usecases/commands/approve-member.command.ts`

#### [item] 主催者がメンバー申請を却下する
- **Labels**: `context:community` `feature:membership` `type:scenario` `status:done`
- **Who**: 主催者（Organizer）
- **What**: PENDING 状態の加入申請レコードを物理削除する。
- **Why**: 不適合な加入申請を取り下げ、状態の持ち越しを避ける。
- **受け入れ基準**:
  - [x] 対象メンバーが存在しない場合 `MemberNotFoundError` を返す
  - [x] 却下により `CommunityMember` レコードが物理削除される
  - [x] `MemberRejected` が発火される
- **DML**: SCENARIO `主催者がメンバー申請を却下する`, CMD `RejectMember`, EVT `MemberRejected`
- **関連コード**: `backend/src/community/usecases/commands/reject-member.command.ts`

#### [item] メンバーがコミュニティから脱退する
- **Labels**: `context:community` `feature:membership` `type:scenario` `status:done`
- **Who**: メンバー（Member）
- **What**: 自身の `CommunityMember` レコードを物理削除する。OWNER は脱退不可。
- **Why**: メンバーの離脱を安全に処理しつつ、オーナー不在によるコミュニティ運営不能を防ぐ。
- **受け入れ基準**:

  - [x] `MemberLeft` が発火される
- **DML**: SCENARIO `メンバーがコミュニティから脱退する`, CMD `LeaveCommunity`, EVT `MemberLeft`

#### [item] システムがキャンセル待ち先頭を繰り上げる
- **Labels**: `context:participation` `feature:cancellation` `type:scenario` `status:todo`
- **Who**: システム（System）— `PromoteFromWaitlist` POLICY 起動
- **What**: `NextWaitlistedParticipation` で取得した WAITLISTED 先頭を APPROVED に繰り上げる。
