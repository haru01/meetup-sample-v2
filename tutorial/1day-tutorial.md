## 09:30 - 09:45 基礎知識：AIエージェントとは何か（15分）

### チャット型との違い

まず、普段使っているチャット型AIとエージェント型AIの違いを押さえる。

| | チャット型（例：Claude.ai） | エージェント型（例：Claude Code） |
|--|--------------------------|---------------------------------------|
| **やりとり** | 1問1答 | ゴール提示 → 計画レビュー → 自律実行 → 作成物レビュー |
| **文脈の保持** | 人間がスレッドを管理 | ファイル（CLAUDE.md 等）が文脈を保持 |
| **ツール利用** | なし〜限定的 | ファイル読み書き・コマンド実行・ブラウザ操作など |
| **得意なこと** | 質問への回答、アイデア出し | ゴールに向かって試行錯誤し達成する |

> **一言で言うと**: チャット型は「聞けば答える」、エージェント型は「任せれば動く」。

### エージェントへの仕事の依頼の仕方

エージェントは「作業指示」を受けて動くのではなく、**ゴールを渡されると自律的に考えて動く**。
うまく依頼できると、エージェントは既存コードを調査し、計画を立て、実装し、検証まで一気に進める。

依頼に含めるべき要素は4つ：

| 要素 | 問い | 役割 |
| --- | --- | --- |
| **ゴール** | 何が達成されていればOKか？ | エージェントが向かう方向を決める |
| **制約** | 守らなければならないルールは何か？ | 既存設計・パターンから逸脱しないようにする |
| **具体例** | どんな状態になっていれば合格か？ | 実装のあいまいさをなくす |
| **検証手段** | どうすれば正しいと確認できるか？ | エージェントが自律的に合否を判断できるようにする |

**ゴール** — 「何をするか」ではなく「何が達成されていればOKか」を伝える

```text
✕ 「EventControllerにPATCHエンドポイントを追加して、Prismaでstatusを更新して返して」
○ 「主催者が DRAFT のイベントを PUBLISHED に公開できるAPIを実装して」
```

「何をするか」を指示すると、エージェントは言われた通りにしか動かない。
「何が達成されていればOKか」を伝えると、エージェントは既存コードを調べ、最善の手段を**自律的に**選ぶ。

**制約** — プロジェクトのルール・設計方針を渡す

エージェントは何も言わなければ「一般的な実装」をしようとする。
プロジェクト固有のパターン（Result型・Command構造・エラー設計など）は**ハーネスか参考ファイルで示す**。

```text
参考
- create-event.command.ts が実装パターンの参考になる（同コンテキストの実装済みコマンド）
- approve-member.command.ts がステータス遷移の参考になる（PENDING → ACTIVE）
```

**具体例（受け入れ条件）** — 「合格の定義」を箇条書きで渡す

受け入れ条件があると、エージェントはそれをテストケースとして使い、実装の精度が上がる。

```text
受け入れ条件
- [ ] `status=DRAFT` のときのみ公開できる
- [ ] すでに公開済みの場合 `EventAlreadyPublishedError` を返す
- [ ] 公開後は `status=PUBLISHED` に遷移する
- [ ] `EventPublished` が発火される
```

**検証手段** — エージェントが自律的に合否を判断できるようにする

「実装して」だけだと、エージェントはコードを書いて終わりにしがちになる。
「テストを通してから報告して」と伝えると、エージェントは自分で検証し、失敗したら修正して再試行するループを**自律的に**回す。

```text
品質ゲートを通してから完了報告して：
  cd backend && npm run test:coverage
  cd backend && npm run review
```

### ハーネス：エージェントが期待通りに動く環境

エージェントに「ゴールを示す」だけでは不十分なことがある。
プロジェクトのルール・構造・品質基準などを**ファイルや実行コマンドとして整備**しておくことで、エージェントが自律的に正しく動ける。

この「エージェントが期待通りに動くための環境整備」を**ハーネス**と呼ぶ。

```text
ハーネスの例：
- CLAUDE.md,AGENTS.md → 技術スタック・ディレクトリ構造・命名規約・実行コマンド・品質基準などを宣言
- docs/conventions/   → テストガイドライン・実装パターン
- docs/decision-logs/ → 設計上の意思決定記録（ADR）
- .claude/settings.json → Claude Code の権限・hook・環境変数を宣言（Edit/Write 後に ESLint 自動実行など）
- テスト・Lint         → 品質の自動チェック
```

> **今日のポイント**: 午前でハーネスの恩恵を体感し、午後でハーネスがない世界を体験する。最後にAIと一緒にハーネスを作り直す。

## 09:45 - 10:35 ① コードリーディング：AIに設計指針を解説させる（50分）

### ねらい

実践に入る前に、**AIエージェントにこのプロジェクトの設計と実装パターンを解説させる**。
自分で全ファイルを読むのではなく、「AIに聞く」ことでプロジェクトの全体像を素早く掴む。同時に、AIが**コードを読んで正確に説明できる**ことも体感する。

> **問い**: 初めてのプロジェクトに入ったとき、AIに何を聞けば効率的にキャッチアップできるか？

### 進め方

リポジトリルートで Claude Code を起動し、以下のプロンプトを順番にAIに投げる。
AIの回答を見ながら、参加者同士で「なるほど」「ここはどういう意味？」と対話する。

```bash
cd meetup-sample-v2
claude
```

### Step 1：event コンテキストの全体像を聞く（10分）

今日実装する event コンテキストの構造を把握する：

```text
backend/src/event/ のコードを読んで、このコンテキストの全体像を解説して。
- ディレクトリ構造とレイヤー構造
- 各レイヤーの依存方向を図で示して
- 今日これから実装する PublishEvent は、どこに何を追加すればよいか
```

**AIの回答で確認すべきポイント**:

- `controllers/` → `usecases/commands/` → `models/` の依存方向（外から内へ）
- `shared/` の Result型・エラー型を共通で使っている
- `CreateEvent` が実装済みで、`PublishEvent` は未実装であることを把握する

> **ファシリテーターヒント**: AIが依存方向の図を出したら、「なぜこの方向？逆だと何が困る？」と問いかけると理解が深まる。

### Step 2：実装済みの Command を題材に実装パターンを学ぶ（20分）

今日の課題と同じ event コンテキストの実装パターンを理解する：

```text
backend/src/event/usecases/commands/create-event.command.ts を起点に、
「主催者がイベントを作成する」シナリオの実装パターンを呼び出し順に解説して。
Result 型・エラー型・イベント発行のパターンも合わせて教えて
```

次に、ステータス遷移の参考として community コンテキストを見る：

```text
backend/src/community/usecases/commands/approve-member.command.ts を読んで、
ステータス遷移（PENDING → ACTIVE）の実装パターンを解説して。
今日実装する PublishEvent（DRAFT → PUBLISHED）はこのパターンに近い。
どこが共通で、どこが違うか整理して
```

**確認したいポイント**:

- `createXxxCommand()` 関数がファクトリとして依存を受け取る構造
- `repository.findById()` → ドメイン関数 → `repository.save()` → `eventBus.publish()` の4ステップ
- `err({ type: 'SomeError' })` の Discriminated Union パターン

### Step 3：CLAUDE.md と規約からプロジェクトのルールを把握する（15分）

```text
@CLAUDE.md @docs/conventions/ を読んで、このプロジェクトのコーディング規約の中で
「これを知らずに実装すると間違えやすいポイント」を3つ挙げて
```

（時間を置く。分からない用語はさらに調べる。）

> **ファシリテーターヒント**: `.claude/hooks/` の各 hook も触れてもらうと、「ハーネスが自動で何をやっているか」が可視化される。

### Step 4：ペアシェア（5分）

> **「AIの解説を聞いて、一番印象に残ったパターンは何ですか？」**

- AIに聞いてみて「わかりやすかった」「ここはもっと深掘りしたい」と感じた点
- 「自分で読むより速いか、遅いか？」 — AIにコード解説を頼む体験の率直な感想

### ハンズオン：PublishEvent をプランモードで実装

**今日実装するシナリオ:**

`CreateEvent` でイベントが DRAFT 状態で作成される。次のステップとして、主催者が DRAFT イベントを PUBLISHED に遷移させ、参加申込を受け付け可能にする API を実装する。

**Step 1: 事前確認**

```bash

./scripts/docker-dev.sh up
./scripts/docker-dev.sh install   # 依存関係インストール
./scripts/docker-dev.sh test      # テストが通ることを確認（全グリーンを確認）
```

main ブランチにいることを確認する（作業ブランチが残っていれば stash しておく）：

```bash
git branch        # main であることを確認
git status        # 未コミット変更がないことを確認
```

**Step 2: プランモードで計画を立てる**

Claude Code を起動する：

```bash
claude
```

すでに起動中であればセッションを新しくする（`/clear` コマンドを入力するか、ターミナルを新しく開いて `claude` を実行する）。

起動後、**Shift+Tab** を押してプランモードに切り替える（画面左下に `plan` と表示されたことを確認）。

プランモードで以下を入力する：

```text
次のbackendの実装計画を立てて。疑問点があれば質問して。

## 主催者がイベントを公開する（PublishEvent）

概要：主催者が DRAFT 状態のイベントを PUBLISHED に遷移させ、
      参加申込を受付可能にする。
      event コンテキスト（backend/src/event/）に実装する。

受け入れ条件
- [ ] `status=DRAFT` のときのみ公開できる
- [ ] すでに公開済み（PUBLISHED / CLOSED / CANCELLED）の場合 `EventAlreadyPublishedError` を返す
- [ ] 公開後は `status=PUBLISHED` に遷移する
- [ ] `EventPublished` が発火される

参考
- 同じ event コンテキストの create-event.command.ts が実装パターンの参考になる
- ステータス遷移は community コンテキストの approve-member.command.ts が参考になる
```

エージェントが既存コードを調査し、実装計画を提示する。
計画を確認し、問題なければ承認して次へ進む。

> **ポイント**: プランモード中はエージェントがファイルを変更しない。計画に納得してから承認することで、想定外の変更を防げる。

**Step 3: 実装**

計画を承認すると、エージェントがコード・テストを生成する。

観察ポイント：

- 新しいファイルはどこに配置されたか？（`event/usecases/commands/publish-event.command.ts` など）
- エラー型はどこに追加されたか？（`event/errors/event-errors.ts`）
- `composition.ts` の依存注入が更新されているか？
- 受け入れ条件がテストでカバーされているか？

**Step 4: 検証（品質ゲート）**

改めて、以下を実行して品質ゲートを通す
（AIエージェントが自動で実行するはずだが改めて）：

```bash
./scripts/docker-dev.sh bash -c "cd backend && npm run test:coverage"
./scripts/docker-dev.sh bash -c "cd backend && npm run review"
```

**Step 5: 成果をコミットする**

品質ゲートが通ったら、作業をコミットしておく：


もしくは Claude Code にコミットさせる：

### つまずきやすいポイント
