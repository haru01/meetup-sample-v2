# Claude Code クイックスタート — MTP-011 イベント公開で学ぶ AI 駆動開発

プログラミング経験がある方が Claude Code を使って、**ストーリー1つをフルスタック（API・UI・E2E）で実装する流れ**を体験するチュートリアルです。

題材: [MTP-011 イベント公開](../docs/stories/meetup-todo.md#mtp-011-イベント公開)

所要時間: 約 60 分

> **前提知識**: MTP-010（イベント作成）は実装済みです。MTP-010 のコード（モデル・UseCase・Controller・フォーム・E2E）が今回の実装パターンの参考になります。

---

## 前提

- Claude Code がインストール済み（[公式ガイド](https://code.claude.com/docs/en/getting-started)）
- このリポジトリを clone 済み
- Docker Desktop が起動済み

```bash
git clone <repository-url>
cd meetup-sample
./scripts/docker-dev.sh up
./scripts/docker-dev.sh install
```

## 1. まずコードベースを理解する（5分）

実装を始める前に、Claude Code に既存コードの構造を聞いてみましょう。

### やること

Claude Code を起動し、以下のような質問をする:

```
claude
```

```
このリポジトリの構成を教えて。backend の community コンテキストはどういうレイヤー構造になっている？
MTP-010 イベント作成はどのように実装されている？
```

### コツ

- **Plan Mode（Shift+Tab）** で聞くと、コードを変更せず安全に調査できる
- `@docs/stories/meetup-todo.md` のようにファイルを指定すると、ピンポイントでコンテキストを渡せる
- 「シニアエンジニアに聞くように質問する」のが公式推奨

### 確認ポイント

- `controllers/` → `usecases/` → `models/` のレイヤー構造を把握できたか
- `Result<T, E>` パターン、Branded Types、Zod スキーマの存在を確認できたか
- MTP-010 のイベント作成がどのファイルで実装されているか把握できたか
- フロントエンドの `pages/` / `components/` / `hooks/` の構成を理解できたか

## 2. AI と計画を壁打ちする（15分）

いきなりコードを書かせるのではなく、まず「何をどう作るか」を AI と一緒に考えます。

### やること

MTP-011 のストーリーを渡して、実装計画を相談する:

```
@docs/stories/meetup-todo.md の MTP-011 イベント公開を実装したい。
既存の MTP-010 イベント作成のパターンを参考に、実装計画を立てて。
backend → frontend → e2e の順で。
```

### コツ

- **What（何を作るか）+ Why（なぜそう作るか）+ 既存パターンへの参照**を伝えるのが効果的
- 「MTP-010 を参考に」と言うだけで、AI は既存コードを読んでパターンを踏襲する
- 計画が出たら **Ctrl+G でエディタで開いて編集できる** — 不要なステップを削ったり順番を入れ替えたりしてから承認する

### 計画に含まれるべきもの

| レイヤー | 作成・変更するもの |
|---------|------------------|
| Backend モデル | `event.ts` にステータス遷移関数 `publishEvent()` を追加 |
| Backend UseCase | `publish-event.command.ts` |
| Backend Controller | `event-controller.ts` に PATCH エンドポイント追加、`event-openapi.ts` 更新 |
| Backend エラー | `event-errors.ts` に `PublishEventError` 追加 |
| Event Bus | `EventPublishedEvent` の発行 |
| Frontend | イベント詳細ページに「公開する」ボタン、確認ダイアログ |
| E2E | `event.spec.ts` にイベント公開フローを追加 |

### 壁打ちで聞くと良い質問

計画が出たあと、例えば以下のような質問で深掘りすると計画の精度が上がる:

- 「DRAFT → PUBLISHED のステータス遷移ロジックはモデル層に置く？UseCase に書く？」
- 「EventPublishedEvent のイベントバス発行はどのレイヤーで行う？既存の InMemoryEventBus の使い方は？」
- 「イベント詳細ページはまだない（MTP-014）けど、公開ボタンはどこに配置する？」

> **なぜ計画を壁打ちするのか？**
> 計画なしでいきなり「実装して」と頼むと、AI は全体の整合性を考えずにファイルを作り始めることがある。5分の壁打ちで30分の手戻りを防げる。

## 3. AI にコードを書かせる（25分）

計画を承認したら、実装に移ります。AI出力が安定しない場合、レビューが大変と負荷を感じる場合、AIに慣れてない場合は、一度に全部頼まず、レイヤーごとに進めるのがポイントです。

> **大事な心構え**: CLAUDE.md の品質ゲートやテスト込みの依頼は、AI 出力の最低ラインを保証するガードレールとなります。ただ、それだけで安心せず、**生成されたコードとテストの両方を人が確認し、問題があれば修正を依頼する**ことが欠かせません。

### 3-1. Backend: モデル + UseCase（10分）

```
MTP-011 の backend を実装して。まず models/ と usecases/ から。
event.ts にステータス遷移関数を追加し、publish-event コマンドを作って。
既存の create-event のパターンに合わせて。
```

#### コツ

- **小さく頼んで、確認してから次へ**。「backend 全部作って」より「モデルと UseCase から」の方が軌道修正しやすい
- 生成されたコードの diff を確認する。特に:
  - `Result<T, E>` を返しているか（例外を throw していないか）
  - ステータス遷移ロジックが `models/` にあるか（UseCase はオーケストレーションのみか）
  - `InvalidEventStatus` エラーで不正な遷移を弾いているか

#### テストを実行し、テストコードを確認

AIエージェントが、もしテストを実行してない場合は、テスト実行を改めて依頼します。
```
backend のテストを実行して
```

テストが通っても安心しない。**テスト自体が正しいかを人が確認する**のが重要:

- テストが受入基準を正しく表現しているか（「公開できる」だけでなく「DRAFT 以外からは公開できない」も検証しているか）
- 境界ケースが漏れていないか（例: 既に PUBLISHED のイベントを再公開しようとした場合）
- モックが現実の振る舞いと乖離していないか

問題を見つけたら具体的に修正を依頼する: 「CANCELLED 状態からの公開もテストして」「モックの戻り値が実際の Repository と違う」など。

### 3-2. Backend: Controller + OpenAPI（5分）

```
次に controller と openapi 定義を作って。
PATCH /communities/:communityId/events/:eventId/publish エンドポイントを追加して。
express-openapi-validator で検証されるようにして。
```

### 3-3. Frontend: 公開ボタン + 確認ダイアログ（7分）

```
次に frontend のイベント公開 UI を作って。
DRAFT 状態のイベントに「公開する」ボタンを表示し、確認ダイアログ後に公開する。
EventCreateForm.tsx のスタイルを参考にして。
```

#### コツ

- フロントエンドは **参考ファイルを具体的に指名** すると、スタイルや構造が揃いやすい
- 「Tailwind のクラスは既存コンポーネントと揃えて」と伝えるのも有効

### 3-4. E2E テスト（3分）

```
最後に e2e テストを作って。event.spec.ts のパターンに合わせて。
オーナーが DRAFT イベントを公開し、ステータスが PUBLISHED に変わるフローをテストして。
```

### 途中で期待と違う出力が出たら

1. **Esc で即座に止める** — AI が間違った方向に進んでいるのを見たら、完了を待たずに止める
2. **何が違うか具体的に伝える** — 「違う」ではなく「ステータス遷移ロジックが UseCase に入っている、models/ に移して」
3. **テストが通っていても鵜呑みにしない** — テスト自体が間違っていれば「全テスト通過」は品質を保証しない。テストコードも diff で確認する
4. **2回修正しても直らなければ `/clear` して再出発** — コンテキストが汚れている可能性がある

## 4. AI とコードレビューする（15分）

実装が一通りできたら、コミット前にレビューします。

### 4-1. diff を確認する

```
今回の変更の diff を見せて
```

diff を読んで、以下を自分の目で確認する:

- [ ] ファイルの配置は計画通りか
- [ ] 不要なファイルが生成されていないか
- [ ] import パスは既存のエイリアス（`@shared/`、`@auth/` 等）を使っているか

### 4-2. AI にレビューを依頼する

```
今回の変更をレビューして。以下の観点でチェックして:
- AGENTS.md の Core Patterns に準拠しているか
- Result 型を正しく使っているか
- UseCase がオーケストレーションのみか（ドメインロジックが混入していないか）
- OpenAPI 定義でドメイン制約を重複定義していないか
- ステータス遷移が DRAFT → PUBLISHED のみ許可されているか
```

#### コツ

- **レビュー観点を具体的に指定する** — 「レビューして」だけだと表面的な指摘しか出ない
- AGENTS.md や CLAUDE.md のルールを引用すると、プロジェクト固有の基準でレビューできる
- 指摘が出たらその場で修正を依頼する:「指摘の 1, 3 を修正して」

> **なぜ AI にレビューを頼むのか？**
> 自分で diff を読むのは前提。その上で AI にレビューを頼むと、見落としがちなパターン違反やエッジケースを拾える。AI は「動くか」だけでなく「プロジェクトの規約に合っているか」もチェックできる。

### 4-3. テストで最終確認

```
backend のテストと e2e テストを実行して、すべて通ることを確認して
```

### 4-4. コミットする

すべて通ったら:

```
 /japanese-commit
```

## 振り返り

このチュートリアルで体験した AI 駆動開発のサイクル:

```
探索 → 計画 → 実装 → レビュー → コミット
```

### うまくいくコツまとめ

| フェーズ | コツ |
|---------|------|
| 探索 | Plan Mode で安全に調査。シニアエンジニアに聞くように質問する |
| 計画 | What + Why + 既存パターン参照で伝える。計画を Ctrl+G で編集してから承認 |
| 実装 | レイヤーごとに小さく頼む。diff を確認してから次へ進む |
| 確認 | **テストコード自体を人が確認する**。テスト通過 ≠ 正しい実装 |
| レビュー | 観点を具体的に指定する。AGENTS.md のルールを引用する |
| 軌道修正 | Esc で止める。2回修正して直らなければ `/clear` |

### よくある失敗パターン

| パターン | 対策 |
|---------|------|
| 「全部作って」と丸投げ | レイヤーごとに分割して依頼する |
| diff を見ずに次へ進む | 各ステップで変更内容を確認する習慣をつける |
| テスト通過で安心する | テストコード自体が正しい仕様を検証しているか人が確認する |
| エラーが出たら即 AI に丸投げ | まず自分でエラーメッセージを読み、仮説を持ってから聞く |
| 1セッションに詰め込みすぎ | タスク間で `/clear` してコンテキストをリセットする |

### AI 出力が期待と違うときの処方箋

AI の出力が不安定なとき、「もう一回やって」と繰り返す前に何が原因かを見極める。対策は **プロンプトエンジニアリング**（💬 依頼の伝え方）、**コンテキストエンジニアリング**（📋 AI に渡す情報の管理）、**ハーネスエンジニアリング**（🔧 プロジェクト設定・自動検証の整備）の3軸に分かれる。

**💬 プロンプトエンジニアリング** — 依頼の伝え方で出力を変える

| 症状 | 原因の見立て | 処方箋 |
|------|------------|--------|
| 要件の解釈がズレる | 計画段階での合意が不十分 | 💬 **計画の壁打ちで合意を作る** — 曖昧な部分を質問で潰し、具体的なシナリオで認識を揃える。「実装前にXxxについて質問して」と頼むと AI がインタビュアーになり、考慮漏れを先に洗い出せる |
| 実装の方向性が見当違い | AI が「どう作るか」を推測している | 💬 **具体例を示して依頼する** — 「MTP-010 の create-event のパターンに合わせて」で参考コードを特定する |
| 動くが正しいか判断しづらい | 検証方法が AI に伝わっていない | 💬 **検証手段を伝えて依頼する** — 「Playwright でコミュニティ作成をブラウザで動作確認して」のように期待する確認方法を添える |
| 変更が大きすぎてレビューしきれない | 依頼の粒度が大きすぎる | 💬 **依頼サイズを小さくする** — 「モデルと UseCase から」のようにレイヤー単位で分割する |
| 細切れすぎて進まない・文脈が切れる | 依頼の粒度が小さすぎる | 💬 **依頼サイズを大きくする** — AI が安定して出力できる範囲なら一度にまとめる |

- 💬 **プロンプトエンジニアリング**: What + Why + 具体例で伝える。AI は「何を・なぜ・どのパターンで」が揃うと正確に動ける

**📋 コンテキストエンジニアリング** — AI のワーキングメモリを管理する

| 症状 | 原因の見立て | 処方箋 |
|------|------------|--------|
| プロジェクト規約を何度も破る | AI が規約を認識していない | 📋 **CLAUDE.md / AGENTS.md を更新する** — 「なぜそのパターンか」の理由と禁止事項を明文化する。ただしプロジェクト固有でない一般知識の詳細は書かない（AI が既に知っているためトークンの無駄） |
| CLAUDE.md に書いた指示を無視する | 設定ファイルが長すぎてコンテキストを圧迫 | 📋 **CLAUDE.md を減らす** — 機械的に検証できるルールは Hook や Lint に移し、設定ファイルは「AI が判断に迷うもの」だけに絞る |
| 前回のセッションで伝えたことを忘れる | セッション間でコンテキストが消える | 📋 **MEMORY.md に記録する** — ユーザーの好み・判断・フィードバックを永続化し、コンテキストを超えて継続する |
| 過去の重要な意思決定を無視して作業する | 決定の背景が AI のコンテキストにない | 📋 **意思決定ログを残して参照させる** — ADR 等で「何を・なぜ決めたか」を記録し、関連作業時に参照させる |
| 長いセッションで出力品質が落ちる | コンテキストが肥大化して指示が埋もれている | 📋 **タスク間で `/clear` する** — 1タスク完了ごとにコンテキストをリセットし、CLAUDE.md の指示が常に効く状態を保つ |
| 調査・実装・検証を全部を１つのメインエージェント実施で出力がブレる | コンテキストに情報が混在している | 📋 **サブエージェントで役割を分離する** — 調査・実装・検証を別コンテキストで実行し、それぞれが専念する |
| 同じ種類の作業を毎回違うやり方でやる | 手順が暗黙知のままになっている | 📋 **Skill にワークフローを定義する** — コミット、PR作成、レビュー等の繰り返す作業を Skill 化し、`/skill名` で毎回同じ手順を再現する。Skill は load on demand（`/skill名` で呼ばれたときだけ読み込み）なので CLAUDE.md を肥大化させずに手順を管理できる |

- 📋 **コンテキストエンジニアリング**: AI のコンテキストウィンドウは有限のワーキングメモリ。必要な情報を、必要なタイミングで、必要な量だけ渡す。`/clear`・サブエージェント委譲・MEMORY.md 永続化・Skill の load on demand はすべて「コンテキストを清潔に保つ」手段

**🔧 ハーネスエンジニアリング** — 環境で出力品質を底上げする

コンテキストエンジニアリングの活動に加えて例えば次のようなことを見直す

| 症状 | 原因の見立て | 処方箋 |
|------|------------|--------|
| コードのスタイルや構造がバラつく | 参考にすべき既存コードが不統一 | 🔧 **ソースコードを整える** — 1つのコンテキスト（例: MTP-010）を「参照実装」として一貫させれば、AI はそれに倣う |
| 同じ種類のミスが毎回出る | 単純なルール違反だが、人が毎回指摘するコストが高くなっている | 🔧 **機械的チェックを補強する** — Lint ルール、循環複雑度、レイヤー依存チェック、型チェックで自動で弾く |
| 修正しても次のステップで戻る | Hook がなく、修正が定着しない | 🔧 **Hook で即時フィードバックする** — PostToolUse で整形・型チェック、pre-commit でゲートを設け、逸脱を即座に検出する |

- 🔧 **ハーネスエンジニアリング**: 指示で制御するのではなく環境で制御する。整ったコード・明文化された規約・自動検証が揃えば、AI は正しい出力を出しやすい状態で動ける

> **ポイント**: 1つの症状に対して処方箋を1つ試し、次の依頼で改善したか確認する。一度に複数変えると何が効いたかわからなくなる。

### 知っておくと便利な小技

| コマンド / 操作 | 効果 |
|----------------|------|
| `Esc` | 生成を即座に中断。方向が違うと思ったらすぐ止める |
| `Shift+Tab` | Plan Mode に切り替え。コードを変更せず調査・計画だけさせたいときに |
| `Ctrl+G` | AI の出力（計画など）をエディタで開いて直接編集できる |
| `@ファイル名` | プロンプトにファイルを添付してコンテキストに含める |
| `/clear` | コンテキストをリセット。タスク切り替え時に使う |
| `/compact` | 会話を要約して圧縮。コンテキストが大きくなったが途中で切りたくないときに |
| `/diff` | 現在のセッションで変更されたファイルの差分を表示 |
| `/undo` | 直前のファイル変更を取り消す。意図しない変更をすぐ戻せる |
| `/cost` | 現在のセッションのトークン消費量を確認 |
| `/model` | 使用モデルを切り替え。軽い作業は Sonnet、複雑な設計は Opus など使い分け |
| `/init` | CLAUDE.md のひな形を自動生成。新しいプロジェクトの初期設定に |
| `/btw` のパターン | 本題の依頼とは別件の質問をおこなう。メインのコンテキストウインドウの汚染を防げる |
| `「〜を覚えて」` | MEMORY.md にユーザーの好みや判断を保存。次のセッションでも反映される |

## 次のステップ

- [ルーブリック](./rubric.md) で自分の到達レベルを確認する
- **MTP-012（イベント更新）** を自分で計画 → 実装 → レビューしてみる（MTP-010, MTP-011 の実装パターンを参考にできる）
- MTP-013〜015 のイベント管理ストーリーに順番に取り組む
- CLAUDE.md / AGENTS.md を読んで、このプロジェクトの AI 設定がどう構成されているか理解する
