# Claude Code クイックスタート — 「イベント公開」機能で学ぶ AI 駆動開発

プログラミング経験がある方が Claude Code を使って、**1 つのシナリオを backend API として実装する流れ**を体験するチュートリアルです。（frontend / E2E は扱わず、API レイヤーの「計画 → 実装 → レビュー」に集中します）

題材: [主催者がイベントを公開する](../docs/issues-20260419.md#item-主催者がイベントを公開する)（`docs/issues-20260419.md` の event BC / Feature: イベント企画）

所要時間: 約 60 分

> **前提知識**: `CreateEvent`（イベント作成）は実装済みです。`backend/src/event/usecases/commands/create-event.command.ts` 周辺のコードが、今回の実装パターンの参考になります。
>
> **完全実装版の参照先**: [`snapshot/2026-04-19`](../docs/issues-20260419.md) ブランチに Publish / Close / Cancel / 参加申込 / チェックイン / 通知まで含めた完成版が保全されています。行き詰まったら `git diff main snapshot/2026-04-19 -- backend/src/event/` 等で覗けます（ただし写経ではなく「考え方の参照」に留めるのがおすすめ）。

claude

このリポジトリの構成を教えて。backend の community / event コンテキストはどういうレイヤー構造になっている？

```
community 主催者がコミュニティを作成する を例にbackendのコードを解説して
```

```
バックエンドの実装パターンをポイント解説して
```

### コツ

- **Plan Mode（Shift+Tab）** で聞くと、コードを変更せず安全に調査できる

## 2. AI に探索と計画を依頼する（15 分）

いきなりコードを書かせるのではなく、まず、計画を立てます。

### やること

対象シナリオを渡して、実装計画を相談する:

```
@docs/issues-20260419.md の「主催者がイベントを公開する」item を実装したい。
event BC の既存実装（CreateEvent 一式）をパターン参照しながら、
まずは backend に絞って実装計画を立てて。疑問点あれば質問して
```

### コツ

- **What（何を作るか）+ Why（なぜそう作るか）+ 既存パターンへの参照**を伝えるのが効果的
- 「CreateEvent を参考に」と言うだけで、AI は指定した既存コードを読んでパターンを踏襲する
- 計画が出たら **Ctrl+G でエディタで開いて編集できる** — 不要なステップを削ったり順番を入れ替えたりしてから承認する
- **スコープを絞る** — backend だけに絞ることで、計画の粒度が揃い、レビューしやすくなる。必要ならさらに絞る。

### 計画に含まれるべきもの

| レイヤー | 作成・変更するもの |
|---------|------------------|
| Backend モデル | `event.ts` にステータス遷移関数 `publishEvent()` を追加（DRAFT → PUBLISHED） |
| Backend UseCase | `publish-event.command.ts`（`Result<Event, PublishEventError>` を返す） |
| Backend Controller | `event.controller.ts` に PUT エンドポイント追加、`event-openapi.ts` 更新 |
| Backend エラー | `event-errors.ts` に `PublishEventError`（`EventAlreadyPublished` / `Unauthorized` / `EventNotFound`）を追加 |
| Event Bus（任意） | `EventPublished` ドメインイベントの publish（`shared/domain-events.ts` を再生する形） |

### 壁打ちで聞くと良い質問

計画が出たあと、例えば以下のような質問で深掘りすると計画の精度が上がる:

- 「DRAFT → PUBLISHED のステータス遷移ロジックはモデル層に置く？UseCase に書く？」
- 「`EventPublished` の Event Bus 発行はどのレイヤーで行う？」
- 「`CreateEvent` の受入基準との差分は？」

> **なぜ計画を壁打ちするのか？**
> 計画なしでいきなり「実装して」と頼むと、AI は全体の整合性を考えずにファイルを作り始めることがある。5 分の壁打ちで 30 分の手戻りを防げる。計画承認の前に、代替手段をあげて比較検討したり、実現手段についてわからない点を質問しながら理解を深めるがポイント。モブプログラミングでチームと議論しながら確定させると良い。

## 3. AI にコードを書かせる（20 分）

Section 2 で出た計画を**人がレビューして承認**してから、backend の実装に移ります。**計画レビュー → 承認 → 実装 → 差分確認** のサイクルで回します。計画段階で「どのファイルに何を書くか」が合意できていれば、実装プロンプトは短くて済み、レビューは「計画とズレていないか」に集中できます。本プロジェクトは境界づけられたコンテキストを採用しているので、`event` など bounded context を絞って依頼するのも有効。

> **大事な心構え**: CLAUDE.md / AGENTS.md の品質ゲートやテスト込みの依頼は、AI 出力の最低ラインを保証するガードレールとなります。ただ、それだけで安心せず、**計画・生成コード・テストの 3 つを人が確認し、問題があれば修正を依頼する**ことが欠かせません。

### まず計画をレビューする

実装に入る前に、section 2 で AI が出した計画を**人が読み込む**。以下の観点でチェックする:

- [ ] **レイヤーの配置が正しいか**: ステータス遷移ロジックが `models/` に置かれ、UseCase はオーケストレーションのみになっているか
- [ ] **ファイル名と配置が既存パターンと揃っているか**: `publish-event.command.ts` が `usecases/commands/` 配下で、`create-event.command.ts` の命名に沿っているか
- [ ] **受入基準と対応が取れているか**: `docs/issues-20260419.md` の AC と計画上のテストケースが 1:1 対応しているか
- [ ] **エラー型の粒度が妥当か**: `PublishEventError` に `EventNotFound` / `EventAlreadyPublished` / `Unauthorized` が過不足なく揃っているか

気になる箇所は **Ctrl+G で計画をエディタで開いて直接編集**するか、AI に「XX の部分を YY に変えて」と依頼して修正する。納得できたら承認する。

### 計画を承認して実装させる

計画が十分に詰まっていれば、実装指示は短くて良い。Plan Mode なら計画承認でそのまま実装が走るし、通常モードなら次のような一文で十分:

```
OK
```

### コツ

- **計画外のことが出てきたら Esc で止める** — AI が計画にないファイルを触り始めたら止め、計画側を更新してから再開する
- **コンテキストを絞る** — 境界づけられたコンテキスト (`event` など) を指定すると、AI の探索範囲が絞れて出力が安定する
- **制約は念押し** — AGENTS.md の規約は計画と実装の両方で触れると逸脱しにくい

### diff で確認するポイント

生成されたコードの diff を読んで以下を確認する:

- `Result<T, E>` を返しているか（例外を throw していないか）
- ステータス遷移ロジックが `models/event.ts` にあり、UseCase はオーケストレーションのみか
- `EventAlreadyPublished` エラーで DRAFT 以外からの公開を弾いているか
- `event-openapi.ts` でドメイン制約（min/max/enum）を重複定義していないか

### テストを実行し、テストコードを確認

AI がテストを実行していなければ改めて依頼:

```
backend のテストを実行して
```

```
品質ゲートチェックして
```

テストが通っても安心しない。**テスト内容自体が正しいかを人が確認する** のが重要:

- テストが受入基準を正しく表現しているか（「公開できる」だけでなく「DRAFT 以外からは公開できない」も検証しているか）
- 境界ケースが漏れていないか（例: 既に PUBLISHED のイベントを再公開、作成者以外が公開しようとした場合）
- モックが現実の振る舞いと乖離していないか

問題を見つけたら具体的に修正を依頼する: 「CLOSED 状態からの公開もテストして」「モックの戻り値が実際の Repository と違う」など。

### 途中で期待と違う出力が出たら

1. **Esc で即座に止める** — AI が間違った方向に進んでいるのを見たら、完了を待たずに止める
2. **何が違うか具体的に伝える** — 「違う」ではなく「ステータス遷移ロジックが UseCase に入っている、models/ に移して」
3. **テストが通っていても鵜呑みにしない** — テスト自体が間違っていれば「全テスト通過」は品質を保証しない。テストコードも diff で確認する
4. **2 回修正しても直らなければ `/clear` して再出発** — コンテキストが汚れている可能性がある

## 4. AI とコードレビューする（15 分）

実装が一通りできたら、コミット前にレビューします。

### 4-1. diff を確認する

```
今回の変更の diff を見せて
```

diff を読んで、以下を自分の目で確認する:

- [ ] ファイルの配置は計画通りか
- [ ] 不要なファイルが生成されていないか
- [ ] import パスは既存のエイリアス（`@shared/`、`@event/`、`@community/` 等）を使っているか

### 4-2. AI にレビューを依頼する

```
今回の変更をレビューして。以下の観点でチェックして:
- AGENTS.md の Core Patterns に準拠しているか
- UseCase がオーケストレーションのみか（ドメインロジックが混入していないか）
- Issueの受け入れ基準を満たすか
- パフォーマンスやセキュリティ観点で問題ないか
```

#### コツ

- **レビュー観点を具体的に指定する** — 単に「レビューして」だけだと表面的な指摘しか出ない
- AGENTS.md / CLAUDE.md のルールを引用すると、プロジェクト固有の基準でレビューできる
- 指摘が出たらその場で修正を依頼する:「指摘の 1, 3 を修正して」

> **なぜ AI にレビューを頼むのか？**
> AI にレビューを頼むと、見落としがちなパターン違反やエッジケースを拾える。AI は「動くか」だけでなく「プロジェクトの規約に合っているか」もチェックできる。Code xなど別ツールでレビューする、プルリクで別AIでレビューを受ける、ビルドインの /simplify スキルを使うも有効。


### 4-5. コミットする

すべて通ったら:

```
/japanese-commit
```

## 振り返り

このチュートリアルで体験した AI 駆動開発のサイクル:探索 → 計画 → 実装 → レビュー → コミット

```
主催者がイベントを公開するの実装の作業セッションを振り返り、改善点を挙げて
```

### うまくいくコツまとめ

| フェーズ | コツ |
|---------|------|
| 探索 | Plan Mode で安全に調査。シニアエンジニアに聞くように質問する |
| 計画 | What + Why + 既存パターン参照で伝える。計画を Ctrl+G で編集してから承認 |
| 実装 | 計画レビュー → 承認 → 実装 → diff 確認のサイクルで回す。 |
| 確認 | **テストコード自体を人が確認する**。テスト通過 ≠ 正しい実装 |
| レビュー | 観点を具体的に指定する。AGENTS.md のルールを引用する |
| 軌道修正 | Esc で止める。2 回修正して直らなければ `/clear` |

### よくある失敗パターン

| パターン | 対策 |
|---------|------|
| 「全部作って」と丸投げ | レイヤーごとに分割して依頼する |
| diff を見ずに次へ進む | 各ステップで変更内容を確認する習慣をつける |
| テスト通過で安心する | テストコード自体が正しい仕様を検証しているか人が確認する |
| エラーが出たら即 AI に丸投げ | まず自分でエラーメッセージを読み、仮説を持ってから聞く |
| 1 セッションに詰め込みすぎ | タスク間で `/clear` してコンテキストをリセットする |
| 完成版を丸写しする | `snapshot/2026-04-19` は参照先。自分で設計し、比較で学ぶ |

### AI 出力が期待と違うときの処方箋

AI の出力が不安定なとき、「もう一回やって」と繰り返す前に何が原因かを見極める。対策は **プロンプトエンジニアリング**（💬 依頼の伝え方）、**コンテキストエンジニアリング**（📋 AI に渡す情報の管理）、**ハーネスエンジニアリング**（🔧 プロジェクト設定・自動検証の整備）の 3 軸に分かれる。

**💬 プロンプトエンジニアリング** — 依頼の伝え方で出力を変える

| 症状 | 原因の見立て | 処方箋 |
|------|------------|--------|
| 要件の解釈がズレる | 計画段階での合意が不十分 | 💬 **計画の壁打ちで合意を作る** — 曖昧な部分を質問で潰し、具体的なシナリオで認識を揃える。「実装前に Xxx について質問して」と頼むと AI がインタビュアーになり、考慮漏れを先に洗い出せる |
| 実装の方向性が見当違い | AI が「どう作るか」を推測している | 💬 **具体例を示して依頼する** — 「CreateEvent の create-event.command のパターンに合わせて」で参考コードを特定する |
| 動くが正しいか判断しづらい | 検証方法が AI に伝わっていない | 💬 **検証手段を伝えて依頼する** — 「`supertest` で公開エンドポイントの 200/403/409 を検証して」のように期待する確認方法を添える |
| 変更が大きすぎてレビューしきれない | 依頼の粒度が大きすぎる | 💬 **依頼サイズを小さくする** — 「model だけ」「UseCase だけ」のようにレイヤー単位で分割し、変更対象ファイルを絞る |
| 細切れすぎて進まない・文脈が切れる | 依頼の粒度が小さすぎる | 💬 **依頼サイズを大きくする** — AI が安定して出力できる範囲なら一度にまとめる |

- 💬 **プロンプトエンジニアリング**: What + Why + 具体例で伝える。AI は「何を・なぜ・どのパターンで」が揃うと正確に動ける

**📋 コンテキストエンジニアリング** — AI のワーキングメモリを管理する

| 症状 | 原因の見立て | 処方箋 |
|------|------------|--------|
| プロジェクト規約を何度も破る | AI が規約を認識していない | 📋 **CLAUDE.md / AGENTS.md を更新する** — 「なぜそのパターンか」の理由と禁止事項を明文化する。ただしプロジェクト固有でない一般知識の詳細は書かない（AI が既に知っているためトークンの無駄） |
| CLAUDE.md に書いた指示を無視する | 設定ファイルが長すぎてコンテキストを圧迫 | 📋 **CLAUDE.md を減らす** — 機械的に検証できるルールは Hook や Lint に移し、設定ファイルは「AI が判断に迷うもの」だけに絞る |
| 前回のセッションで伝えたことを忘れる | セッション間でコンテキストが消える | 📋 **MEMORY.md に記録する** — ユーザーの好み・判断・フィードバックを永続化し、コンテキストを超えて継続する |
| 過去の重要な意思決定を無視して作業する | 決定の背景が AI のコンテキストにない | 📋 **意思決定ログを残して参照させる** — ADR 等で「何を・なぜ決めたか」を記録し、関連作業時に参照させる |
| 長いセッションで出力品質が落ちる | コンテキストが肥大化して指示が埋もれている | 📋 **タスク間で `/clear` する** — 1 タスク完了ごとにコンテキストをリセットし、CLAUDE.md の指示が常に効く状態を保つ |
| 調査・実装・検証を全部を 1 つのメインエージェント実施で出力がブレる | コンテキストに情報が混在している | 📋 **サブエージェントで役割を分離する** — 調査・実装・検証を別コンテキストで実行し、それぞれが専念する |
| 同じ種類の作業を毎回違うやり方でやる | 手順が暗黙知のままになっている | 📋 **Skill にワークフローを定義する** — コミット、PR 作成、レビュー等の繰り返す作業を Skill 化し、`/skill 名` で毎回同じ手順を再現する。Skill は load on demand（`/skill 名` で呼ばれたときだけ読み込み）なので CLAUDE.md を肥大化させずに手順を管理できる |

- 📋 **コンテキストエンジニアリング**: AI のコンテキストウィンドウは有限のワーキングメモリ。必要な情報を、必要なタイミングで、必要な量だけ渡す。`/clear`・サブエージェント委譲・MEMORY.md 永続化・Skill の load on demand はすべて「コンテキストを清潔に保つ」手段

**🔧 ハーネスエンジニアリング** — 環境で出力品質を底上げする

コンテキストエンジニアリングの活動に加えて例えば次のようなことを見直す

| 症状 | 原因の見立て | 処方箋 |
|------|------------|--------|
| コードのスタイルや構造がバラつく | 参考にすべき既存コードが不統一 | 🔧 **ソースコードを整える** — 1 つのコンテキスト（例: CreateEvent / CreateCommunity）を「参照実装」として一貫させれば、AI はそれに倣う |
| 同じ種類のミスが毎回出る | 単純なルール違反だが、人が毎回指摘するコストが高くなっている | 🔧 **機械的チェックを補強する** — Lint ルール、循環複雑度、レイヤー依存チェック、型チェックで自動で弾く |
| 修正しても次のステップで戻る | Hook がなく、修正が定着しない | 🔧 **Hook で即時フィードバックする** — PostToolUse で整形・型チェック、pre-commit でゲートを設け、逸脱を即座に検出する |

- 🔧 **ハーネスエンジニアリング**: 指示で制御するのではなく環境で制御する。整ったコード・明文化された規約・自動検証が揃えば、AI は正しい出力を出しやすい状態で動ける

> **ポイント**: 1 つの症状に対して処方箋を 1 つ試し、次の依頼で改善したか確認する。一度に複数変えると何が効いたかわからなくなる。

### 知っておくと便利な小技

| コマンド / 操作 | 効果 |
|----------------|------|
| `Esc` | 生成を即座に中断。方向が違うと思ったらすぐ止める |
| `Shift+Tab` | Plan Mode に切り替え。コードを変更せず調査・計画だけさせたいときに |
| `Ctrl+G` | AI の出力（計画など）をエディタで開いて直接編集できる |
| `@ファイル名` | プロンプトにファイルを添付してコンテキストに含める |
| `/clear` | コンテキストをリセット。タスク切り替え時に使う |
| `/compact` | 会話を要約して圧縮。コンテキストが大きくなったが途中で切りたくないときに |
| `/diff` | 現在のセッションで変更されたファイルの差分を表示 |
| `/undo` | 直前のファイル変更を取り消す。意図しない変更をすぐ戻せる |
