# 90分ファシリテーションガイド

## 0〜10分：フレーミング

以下の3問のみで確認する（余分な質問は加えない）：

```
1. 対象フローは？（例：「登録〜内定」「求人掲載〜採用」「全体」）
2. 主要アクター視点は？（ユーザー / 企業 / 両方）
3. ゴールは？（全体像把握 / マイクロサービス分割 / 要件整理 / その他）
```

ドメイン固有の特性も確認するとよい：
- AIエージェントや自動バッチ処理はあるか？
- 外部システム連携（メール・カレンダー・決済など）はあるか？

## 10〜40分：イベント収集（提示→確認スタイル）

**やること**: 候補リストを先に提示し、ユーザーに確認・修正させる。「どんなイベントがありますか？」とは聞かない。

### 手順

1. `references/domain-starters.md` を参照してドメインに近いスターターを選ぶ
2. 10〜20件の候補イベントをフェーズ別に下書きして提示する

```
─── 〇〇フェーズ ───
[1] 〇〇が完了した
[2] 〇〇が登録された
...
─── △△フェーズ ───
[3] 〇〇が実行された
...
```

3. ユーザーに確認：「このタイムラインに**抜けているイベント**はありますか？修正点は？」
4. 追加・修正を反映 → 合意が取れたら次フェーズへ

### イベント記述ルール
- 過去形・受動態で記述する（例：「〇〇が完了した」「〇〇が通知された」）
- 粒度は「運用上意味のある中粒度」（大きすぎず・細かすぎず）
- 不明点は `[#question]` タグで明示し、推測しない

## 40〜65分：コマンド / ポリシー / アクター マッピング

各イベントに対して以下を特定する：

| 確認項目 | 質問例 |
|---|---|
| コマンド | 「このイベントを引き起こしたコマンド（操作）は？」 |
| アクター | 「誰が・何がそのコマンドを実行した？」 |
| ポリシー | 「どんなルール・条件でそのコマンドが実行されましたか？」 |

**アクター種別を必ず確認する：**
- 人間（求職者・企業担当者・管理者など）
- システム自動（イベントトリガーで自動実行）
- AIエージェント（チャット・推薦・日程調整など）
- スケジューラー（バッチ・定期実行）
- 外部システム（メール・決済・カレンダーなど）

テーブル形式で全体を一括提示し、1ラウンドで確認・修正を求める。

## 65〜80分：コンテキストと集約の導出

### 境界付きコンテキスト候補の見つけ方
- 言語（用語の意味）が変わる境界に注目する
- 責務（誰が何を所有するか）が変わる境界に注目する
- イベントクラスターをグループ化して名前をつける

### 集約候補の見つけ方
- 「一緒に変わるもの」をひとつの集約にする
- 不変条件（常に守られなければならないビジネスルール）を列挙する
- 集約の生成・変更・削除の条件を確認する

## 80〜90分：クローズと次のアクション

### オープンクエスチョンのクローズ
- セッション中に蓄積した `[#question]` をすべて確認する
- 解決したものは `[CLOSED]` + 解決内容でマークする
- 残ったものはオーナーと優先度を記録する

### ドラフト確認（出力前の必須ステップ）
ファイル出力の前に、以下をまとめて提示してユーザーのOKをもらう：
- イベントリスト（フェーズ別）
- コマンド / ポリシー / アクター テーブル
- コンテキスト候補と集約候補

「この内容でよければファイルを出力します」と確認する。

## インタビュースタイル

- 1ターンあたり3〜5問に絞る
- 候補を先に提示してユーザーに確認させる（開放型質問は避ける）
- 不確かな箇所は `[#question]` でマーク；推測・補完しない
- 不明な領域を過剰モデリングしない
