# 90分ファシリテーションガイド

## 0〜10分：フレーミング

以下の3問のみで確認する（余分な質問は加えない）：

```
1. 対象フローは？（例：「登録〜内定」「求人掲載〜採用」「全体」）
2. 主要アクター視点は？（ユーザー / 企業 / 両方）
3. ゴールは？（全体像把握 / マイクロサービス分割 / 要件整理 / その他）
```

ドメイン固有の特性も確認するとよい：
- AIエージェントや自動バッチ処理はあるか？
- 外部システム連携（メール・カレンダー・決済など）はあるか？

## 10〜40分：イベント収集（提示→確認スタイル）

**やること**: 候補リストを先に提示し、ユーザーに確認・修正させる。「どんなイベントがありますか？」とは聞かない。

### 手順

1. `references/domain-starters.md` を参照してドメインに近いスターターを選ぶ
2. 10〜20件の候補イベントをフェーズ別に下書きして提示する

```
─── 〇〇フェーズ ───
[1] 〇〇が完了した
[2] 〇〇が登録された
...
─── △△フェーズ ───
[3] 〇〇が実行された
...
```

3. ユーザーに確認：「このタイムラインに**抜けているイベント**はありますか？修正点は？」
4. 追加・修正を反映 → 合意が取れたら次フェーズへ

### イベント記述ルール
- 過去形・受動態で記述する（例：「〇〇が完了した」「〇〇が通知された」）
- 粒度は「運用上意味のある中粒度」（大きすぎず・細かすぎず）
- 不明点は `[#question]` タグで明示し、推測しない

## 40〜65分：コマンド / ポリシー / アクター マッピング

各イベントに対して以下を特定する：

| 確認項目 | 質問例 |
|---|---|
| コマンド | 「このイベントを引き起こしたコマンド（操作）は？」 |
| アクター | 「誰が・何がそのコマンドを実行した？」 |
| ポリシー | 「どんなルール・条件でそのコマンドが実行されましたか？」 |

**アクター種別を必ず確認する：**
- 人間（求職者・企業担当者・管理者など）
- システム自動（イベントトリガーで自動実行）
- AIエージェント（チャット・推薦・日程調整など）
- スケジューラー（バッチ・定期実行）
- 外部システム（メール・決済・カレンダーなど）

テーブル形式で全体を一括提示し、1ラウンドで確認・修正を求める。

### インフラ系ドメインの BC 昇格判定（通知・スケジューラ・決済・メール等）

通知・バッチ・外部 API 連携が登場したら、**その場で「BC 昇格 vs POLICY 留置」を判定する**。
判定基準は `references/dml-spec.md` の「インフラ系ドメインの扱い」節を参照：

- 通知種類が 1 種類のみ / 状態を持たない → POLICY 留置
- 通知種類が複数 / SENT/FAILED 等の状態を持つ / 監査ログ兼用 → BC 昇格

判定を保留する場合は `[?]` でホットスポットに残す。
**「データモデルはあるが BC 宣言がない」宙吊り状態を避ける**のが本ステップの目的。

## 65〜80分：コンテキストと集約の導出

### 境界付きコンテキスト候補の見つけ方
- 言語（用語の意味）が変わる境界に注目する
- 責務（誰が何を所有するか）が変わる境界に注目する
- イベントクラスターをグループ化して名前をつける

### BC 間依存方向の導出（必須）

各 BC ペアについて以下を 1:1 で決定する：

1. **どちらがどちらを参照するか**（Command 内や Repository から見て、どの BC のモデルをロード／参照するか）
2. **関係タイプは何か**
   - `Customer-Supplier`: 下流が上流の API/モデルを安定的に利用
   - `Conformist`: 下流が上流の形に従う（変換せず利用）
   - `Shared-Kernel`: 両 BC で共有するモデル部分
   - `ACL`（Anti-Corruption Layer）: 下流が上流を変換レイヤで隔離

決定結果を DML の `CONTEXT` 宣言の `UPSTREAM` / `DOWNSTREAM` に反映する（`references/dml-spec.md` 参照）。

### 集約候補の見つけ方
- 「一緒に変わるもの」をひとつの集約にする
- 不変条件（常に守られなければならないビジネスルール）を列挙する
- 集約の生成・変更・削除の条件を確認する

### 集約属性の Zod スキーマ化（必須）

集約候補が固まったら、不変条件を書き出す直前に **Zod スキーマで属性を固める**：

- Branded ID は `<Name>IdSchema` として別途参照する
- 列挙値は `z.enum([...])` で表現（必ず DML の状態遷移と一致）
- `readonly` が必要なフィールドは Zod 型では不要（TypeScript 側で制御）
- 型定義は `z.infer<typeof Schema>` でエクスポート

属性を書くことで **抜け漏れ**（例: `format`・`startsAt`・`endsAt` などの基本フィールド欠落）を防ぐ。

## 80〜90分：クローズと次のアクション

### オープンクエスチョンのクローズ
- セッション中に蓄積した `[#question]` をすべて確認する
- 解決したものは `[CLOSED]` + 解決内容でマークする
- 残ったものはオーナーと優先度を記録する

### ドラフト確認（出力前の必須ステップ）
ファイル出力の前に、以下をまとめて提示してユーザーのOKをもらう：
- イベントリスト（フェーズ別）
- コマンド / ポリシー / アクター テーブル
- コンテキスト候補と集約候補
- 集約ごとの Zod スキーマ
- BC 間依存方向（UPSTREAM / DOWNSTREAM）
- 日本語↔英語用語集（全フロー図ラベルが登録済みか確認）

「この内容でよければファイルを出力します」と確認する。

## インタビュースタイル

- 1ターンあたり3〜5問に絞る
- 候補を先に提示してユーザーに確認させる（開放型質問は避ける）
- 不確かな箇所は `[#question]` でマーク；推測・補完しない
- 不明な領域を過剰モデリングしない
