# Decision Log — 意思決定記録

不可逆またはインパクトの大きい意思決定を `docs/decision-logs/` に記録する。後から「なぜこう決めたのか」を追跡できるようにするためのスキル。

## 対象とする意思決定

すべての決定を記録する必要はない。**元に戻すのが難しい判断**や**影響範囲が広い判断**を対象にする。

| 種類 | プレフィックス | 例 |
|------|---------------|-----|
| アーキテクチャ・技術選定 | `arch` | ORM の採用、認証方式の変更、DB スキーマの大幅変更、新しいアーキテクチャパターンの導入 |
| ビジネス判断 | `biz` | 機能の廃止、課金モデルの変更、対象ユーザーの変更、SLA の設定 |
| プロセス・運用変更 | `proc` | 開発フローの変更、品質基準の変更、デプロイ戦略の変更、チーム構成の変更 |

## プロアクティブな検出

以下のような変更を検出したら、意思決定ログの作成を提案する:

- 新しいライブラリ/フレームワークの導入または入れ替え
- DB スキーマの破壊的変更（カラム削除、型変更など）
- API の破壊的変更（エンドポイント削除、レスポンス形式変更）
- 認証・認可方式の変更
- 既存機能の廃止や大幅な仕様変更
- インフラ構成の変更（Docker構成、CI/CD パイプライン）

提案時は押し付けず確認する: 「この変更は不可逆性が高いため、意思決定ログとして記録することを提案します。記録しますか？」

## ファイル命名規則

```
<type>-<YYYYMMDD>-<slug>.md
```

- `<type>`: `arch` | `biz` | `proc`
- `<YYYYMMDD>`: 決定日（今日の日付）
- `<slug>`: 英語 kebab-case で決定内容を簡潔に表す

**例:**
- `arch-20260404-adopt-prisma-orm.md`
- `biz-20260404-deprecate-free-plan.md`
- `proc-20260404-adopt-trunk-based-dev.md`

## テンプレート

```markdown
# <決定タイトル>

日付: YYYY-MM-DD
種類: arch | biz | proc
ステータス: 承認済み | 提案中 | 廃止

## コンテキスト

なぜこの決定が必要になったのか。背景にある課題や制約を説明する。

## 決定

何を決めたのか。簡潔に記述する。

## 理由と代替案

**採用した理由:**
- ...

**検討した代替案:**
- **代替案A**: 概要 → 選ばなかった理由: ...
- **代替案B**: 概要 → 選ばなかった理由: ...

## 影響

**ポジティブ:**
- ...

**ネガティブ・リスク:**
- ...
```

代替案のセクションでは、各代替案について**選ばなかった理由を必ず明記する**。「なぜそちらを選ばなかったのか」が将来の判断材料になるため。

## 手順

### ステップ1: インタビュー

ログを書き始める前に、ユーザーに以下の質問をして情報を集める。会話の中で既に答えが出ている項目はスキップしてよい。一度にすべて聞かず、足りない情報だけを聞く。

1. **何を決めたのか？** — 決定の内容を一言で
2. **なぜその決定が必要になったのか？** — 背景にある課題や制約
3. **他にどんな選択肢を検討したか？** — 代替案とそれぞれを選ばなかった理由
4. **この決定でどんな影響があるか？** — 良い面と、リスクや懸念

ユーザーの回答が十分に揃ったら、ステップ2に進む。情報が不足している場合は追加で質問する。特に「代替案を選ばなかった理由」は将来の判断材料になるため、曖昧なら深掘りする。

### ステップ2: ログ作成

1. `docs/decision-logs/` ディレクトリがなければ `mkdir -p` で作成する
2. 既存ファイルを `ls docs/decision-logs/` で確認し、同じ日付・同じトピックの重複がないか確認する
3. 種類（`arch` / `biz` / `proc`）を判断する
4. インタビューで集めた情報をもとに、テンプレートに沿って日本語で内容を記述する
5. ファイルを `docs/decision-logs/<type>-<YYYYMMDD>-<slug>.md` に保存する
